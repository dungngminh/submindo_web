# AGENTS.md

## Project Overview

Submindo Web is a static Astro 6 landing site for the Submindo subscription tracking app. It uses npm, Astro components, TypeScript in `.astro` frontmatter/client scripts, and a custom neobrutalist design system.

Key paths:

- `src/pages/`: route pages (`index.astro`, `privacy.astro`, `terms.astro`)
- `src/components/`: reusable page sections
- `src/layouts/Layout.astro`: shared document shell
- `src/styles/global.css`: design tokens and shared component styles
- `src/lib/brands.ts`: brand metadata fetch/cache and icon-to-image mapping
- `public/`: static assets copied directly into production output
- `public/brands/`: brand logo `.webp` assets

## Setup Commands

- Install dependencies: `npm install`
- Start development server: `npm run dev`
- Build production site: `npm run build`
- Preview production build: `npm run preview`

The project requires Node `>=22.12.0` per `package.json`.

## Development Workflow

- Use npm. Keep `package-lock.json` in sync when dependencies change.
- Astro dev server defaults to `localhost:4321`.
- Production output is generated into `dist/`.
- Do not edit `dist/` directly; it is generated by `npm run build`.
- Source maps are explicitly disabled in `astro.config.mjs` with `vite.build.sourcemap: false`.
- The sitemap integration writes `sitemap-index.xml` and sitemap files during build.

## Brand Assets

Brand ordering, names, colors, and icon slugs come from the remote JSON in `src/lib/brands.ts`:

```ts
https://raw.githubusercontent.com/dungngminh/submindo_asset/refs/heads/master/type.json
```

The site maps each remote `icon` slug to a local image path through `iconToImg` in `src/lib/brands.ts`. When adding or renaming a brand image:

- Add the `.webp` file under `public/brands/`.
- Update `iconToImg` in `src/lib/brands.ts`.
- Verify the slug matches the remote `icon` value.
- Run `npm run build` to confirm the build can fetch the remote JSON and render the brand marquee.

Because `public/` files are served as public URLs, do not place secrets or private mapping data there.

## Code Style

- Follow existing Astro component patterns: frontmatter first, markup second, component-scoped `<style>`, then minimal client `<script>` when needed.
- Use strict TypeScript-compatible code; `tsconfig.json` extends `astro/tsconfigs/strict`.
- Keep shared design tokens in `src/styles/global.css`.
- Prefer existing CSS variables such as `--primary`, `--text-primary`, `--border`, `--shadow`, and spacing tokens before adding new values.
- Keep the visual style consistent with the current neobrutalist system: solid black borders, hard shadows, compact radii, and high-contrast surfaces.
- Store static images in `public/` and reference them with root-relative paths such as `/brands/img_ms365.webp`.

## Testing and Verification

There is no dedicated test or lint script currently. Use these checks before finishing changes:

- Run `npm run build` for type checking, Astro compilation, and static generation.
- Run `npm run preview` when visual or routing behavior needs manual verification.
- For brand-related changes, inspect the built HTML or preview page to confirm the expected image URL appears.

If `npm run build` fails with `getaddrinfo ENOTFOUND raw.githubusercontent.com`, the build environment cannot reach the remote brand JSON. Rerun in an environment with network access.

## Build and Deployment

- Production build command: `npm run build`
- Build output: `dist/`
- Site URL in config: `https://submindo-web.vercel.app`
- `robots.txt` references `https://submindo-web.vercel.app/sitemap-index.xml`

Before deployment, confirm `npm run build` succeeds and no unwanted generated files are committed.

## Agent Notes

- Do not revert unrelated local changes; this repo may have active edits.
- Keep changes scoped to the requested component/config/assets.
- Avoid adding new tooling unless the task requires it.
- If changing public assets, remember browsers and CDNs may cache old files. Renaming an asset or changing the reference path is the most reliable cache-busting option.
